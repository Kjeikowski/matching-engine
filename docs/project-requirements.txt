Concurrent Trade Matching Platform - Updated Project Roadmap

Goal:
Build a high-performance, concurrent backend trading system in C++ that demonstrates backend architecture, design patterns, concurrency, performance engineering, and structured service-to-service messaging using RPC and Protocol Buffers.

Core Requirements:
- C++ heavy implementation
- Backend-oriented architecture
- Builder Pattern at least once
- Factory Pattern at least once
- Follow Dependency Inversion Principle (DIP)
- Concurrency as a core part of the design
- Performance benchmarking as a required deliverable
- RPC + Protocol Buffers as a required communication layer
- Unit, integration, concurrency, and performance testing
- Persistence layer behind interfaces
- Professional documentation

Important Design Choice for Builder Pattern:
Use the Builder Pattern for an EngineConfig / ServiceConfig object.
Why:
- In real backend systems, configuration objects are often the most natural and realistic place to use Builder
- They usually have many optional settings
- It keeps setup code readable and maintainable
- It feels more production-like than forcing Builder onto a simple object

Example builder target:
- EngineConfig
  - worker thread count
  - queue capacity
  - persistence mode
  - logging level
  - metrics enabled/disabled
  - matching strategy
  - RPC port
  - database connection string

You can still optionally use Builder for Order creation too, but EngineConfigBuilder should be the primary required Builder usage.

PHASE 2: Backend Architecture & DIP (1-2 weeks)
--------------------------------------------------
- Introduce layered architecture:
  - RPC/API layer
  - Service layer (OrderService)
  - Repository interfaces
  - Domain layer
  - Infrastructure layer

- Create interfaces:
  - IOrderRepository
  - ITradeRepository
  - IEventPublisher (optional)
  - IClock (optional for testability)

- Implement:
  - InMemoryOrderRepository
  - InMemoryTradeRepository

- Refactor MatchingEngine and services to depend on interfaces only

Deliverable:
- Clean architecture with dependency inversion
- System works end-to-end in memory
- Clear separation between transport models and domain models

--------------------------------------------------
PHASE 3: Protocol Buffers + RPC Layer (1-2 weeks)
--------------------------------------------------
- Define protobuf schemas for:
  - SubmitOrderRequest
  - SubmitOrderResponse
  - CancelOrderRequest
  - CancelOrderResponse
  - TradeMessage
  - OrderBookSnapshot
  - MetricsResponse

- Add gRPC service definitions such as:
  - SubmitOrder
  - CancelOrder
  - GetOrderBook
  - GetTrades
  - GetMetrics

- Generate C++ code from .proto files
- Implement gRPC server
- Map protobuf transport messages to internal domain objects
- Keep matching engine decoupled from protobuf-generated classes

Deliverable:
- Working gRPC interface using protobuf
- Strong service contract for backend communication

Note:
- Optional REST endpoints can still be added later for easy curl/Postman testing
- gRPC + protobuf is the required communication layer

--------------------------------------------------
PHASE 4: Concurrency (2 weeks)
--------------------------------------------------
- Implement thread-safe queue
- Create producer-consumer pipeline:
  - RPC/API threads push orders into inbound queue
  - Matching worker thread(s) consume orders
  - Execution events queued for persistence and metrics

- Add:
  - ThreadPool
  - Worker threads
  - Synchronization primitives (mutex, condition_variable, atomics)

- Ensure thread safety in:
  - order submission path
  - order book access
  - metrics aggregation

Deliverable:
- Concurrent system processing multiple orders safely
- Concurrency tests validating correctness under parallel submission

--------------------------------------------------
